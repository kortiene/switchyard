## Design Context

> Scope note: this project's only "frontend" is the **ADW Cockpit**, a Pi TUI
> extension (`.pi/extensions/adw-cockpit/index.ts`) rendered with terminal
> primitives (`Container`, `Text`, `DynamicBorder`) and theme **tokens**
> (`theme.fg`/`theme.bg`/`theme.bold`/`theme.inverse`). There is no web stack,
> so the web-specific guidance (fonts, OKLCH, clamp, gradients, motion) does
> not apply; the transferable principles do: bold hierarchy, a dominant color
> with rare sharp accents, spatial rhythm, intentional structure, and a
> distinctive point of view that is not generic.

### Users
Developers/operators driving the ADW SDLC pipeline from inside Pi. They keep the
cockpit on-screen while working and glance at it ambiently; when something needs
attention they drill in with `/adw-runs`. The cockpit is strictly read-only — it
observes config, git, and `agents/*/state.json`; it never owns git/forge.

### Brand Personality
Instrument-grade · deliberate · operational. The literal product name is
"Cockpit", so the metaphor is a **mission-control dashboard**: every readout is
real signal, calm until something needs attention, confident rather than
decorative.

### Aesthetic Direction
A dark **mission-control dashboard** built entirely from terminal capabilities
(modelled on the reference screen the maintainer supplied):
- **Numbered, titled, box-drawn panels** (`1. OVERVIEW`, `2. LATEST RUN`,
  `3. PIPELINE`) drawn with `┌─┐ │ └─┘` to the live editor width; the cockpit is
  a custom component so it can size real boxes (not a fixed `string[]`).
- **`key : value` rows** with a dim, padded key column against status-coloured
  values — the dense, aligned readout style of the reference.
- **Status dots + green completion meters**: phases render as a `●`/`◉`/`○`
  status-dot grid (done / current / pending) under a `▕███░░▏ n/total` bar.
- A heading line (`ADW COCKPIT … updated HH:MM:SS  ● live`) and right-aligned
  title tags (a `PR` badge, the `n/total` count) echo the reference chrome.
- **Responsive**: two panels sit side by side when wide, stack when narrow; the
  phase grid uses 2–4 columns by width.
- Theme-respecting colour: `accent` headings, `success` (green) for done/healthy,
  `warning` (amber) for in-progress/missing, `error` for alerts, `border` for
  the frames, `dim`/`muted` as the ground. Never hardcode colours; always go
  through theme tokens so it honours the user's terminal theme.

### Design Principles
1. **Every glyph is signal.** No decoration that doesn't encode real run state.
2. **Bold via hierarchy, not noise.** Heavy header + dim labels + bold values;
   one highlighted current-phase chip, not a rainbow.
3. **Honest instrumentation.** Show all phases; mark untracked kernel wrappers
   and skipped gates truthfully (matches what the orchestrator persists).
4. **Bolder, not bigger.** Amplify impact within roughly the same vertical
   footprint; the cockpit sits above a working editor.
5. **Theme-native.** Colours/backgrounds come from theme tokens and degrade
   gracefully across light/dark terminal themes.
